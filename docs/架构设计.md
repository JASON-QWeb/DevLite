# DevLite 架构设计

## 产品定位

DevLite 不是开发者专用的 F12 替代品，而是普通用户也能使用的网页问题诊断和可视化改样工具。

核心目标：

1. 普通用户不用打开开发者工具，也能生成可交给开发者的错误报告。
2. 前端开发者可以直接在页面上调整 CSS，并导出为可实现的任务说明。
3. 默认本地分析，减少隐私风险和上架审核压力。

## 模块职责

### popup

popup 是主要操作入口，负责：

- 启动和停止诊断
- 启动元素选择器
- 展示错误和样式修改统计
- 生成 Markdown 报告
- 复制 AI Prompt / Markdown / JSON 导出
- 展示 AI 发送前预览
- 调用用户配置的 AI API Key

### options

options 负责长期配置：

- 是否采集 response 摘要
- response 摘要最大长度
- 慢请求阈值
- 额外脱敏字段
- AI 模式
- AI 服务商、模型和 API Key

API Key 通过 `chrome.storage.local` 保存在用户本地浏览器中。

### background service worker

background 是数据和能力中心：

- 按 tab 管理诊断会话
- 注入 content script 和 injected script
- 聚合诊断事件
- 聚合样式修改记录
- 执行脱敏
- 执行本地规则分析
- 生成报告和 Prompt
- 在用户确认后调用 AI 服务

### content script

content script 运行在 Chrome 扩展隔离环境中：

- 接收 injected script 通过 `window.postMessage` 发来的事件
- 捕获 DOM 资源加载失败
- 记录用户点击
- 渲染页面右侧诊断栏入口
- 渲染元素选择高亮层
- 渲染可拖动 CSS 编辑浮层
- 应用临时 inline style
- 记录 before / after 样式修改

content script 不能直接访问页面 JavaScript 运行环境，因此不负责 hook 页面原生的 `fetch` 和 `XMLHttpRequest`。

### injected script

injected script 通过 `chrome.scripting.executeScript` 注入到页面 `MAIN` world：

- hook `window.fetch`
- hook `XMLHttpRequest`
- hook `console.error`
- 监听 `window.error`
- 监听 `unhandledrejection`

采集到的数据通过 `window.postMessage` 发给 content script，再由 content script 发给 background。

## 数据流

```text
用户点击 popup 开始诊断
        ↓
background 注入 content.js 和 injected.js
        ↓
content script 启动诊断并通知 injected script
        ↓
injected script 采集 JS / Network / console 事件
        ↓
content script 采集资源失败、点击路径、CSS 修改
        ↓
background 聚合、脱敏、分析
        ↓
popup 展示统计、报告、导出内容
```

## 样式修改记录

每次选择元素后，DevLite 会记录：

- selector
- DOM 路径
- 元素标签和 class
- 元素文本摘要
- 修改前 computed style
- 修改后 CSS 属性
- 当前视口尺寸

这些记录不会假设源码文件路径。导出 AI Prompt 时，会明确要求模型根据 selector、DOM 路径和文本在项目中定位真实组件。

## 本地规则分析

当前内置规则包括：

- 401：登录态、Token 或鉴权问题
- 403：权限或 CSRF 问题
- 404：接口路由、资源路径或发布路径问题
- 5xx：后端服务异常
- CORS：跨域配置问题
- Mixed Content：HTTPS 页面加载 HTTP 资源
- ChunkLoadError：前端资源版本或缓存问题
- Cannot read properties：空值保护不足或接口结构变化

这些规则不依赖 AI，可离线运行。
